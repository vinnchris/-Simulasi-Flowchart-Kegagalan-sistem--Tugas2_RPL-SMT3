ANALISIS KEBUTUHAN

Kebutuhan Fungsional (FR)

Skenario Kasus (Soal 1): 1. Identifikasi minimal 5 kebutuhan fungsional yang harus dimiliki oleh sistem marketplace ini.

Manajemen User & Pesanan

  • FR-01 Registrasi Akun: Sistem harus mampu meregistrasi dan menyimpan data profil pelanggan baru ke database.
  • FR-02 Katalog & Keranjang: Sistem harus menyediakan fungsi pemilihan barang multi-vendor dan penyimpanan draf belanja.
  • FR-03 Pembuatan Pesanan: Menyimpan record pesanan ke dalam basis data transaksi utama secara terstruktur.

Transaksi & Layanan Logistik

  • FR-04 Integrasi Gateway: Mekanisme tautan API pihak ketiga untuk otorisasi dan cek saldo bank secara real-time.
  • FR-05 Resi & Notifikasi: Sistem otomatis mengirim pemberitahuan pesanan baru ke penjual serta update nomor tracking kurir ke pembeli.
  • FR-06 Resolusi & Ulasan: Fitur input rating produk dan pembuatan tiket keluhan/klaim refund kepada admin.
ANALISIS KEBUTUHAN

Kebutuhan Non-Fungsional (NFR)

Skenario Kasus (Soal 1): 2. Sebutkan minimal 3 kebutuhan non-fungsional yang penting untuk sistem ini.

Atribut Kualitas Utama

  • Security (Keamanan Data): Menggunakan enkripsi SSL/TLS tingkat tinggi untuk melindungi lalu lintas data finansial sensitif pengguna agar terhindar dari peretasan.
  • Performance & Real-Time: Subsistem sinkronisasi ke payment gateway eksternal wajib merespons status dalam hitungan milidetik guna menghindari pemutusan session.
  • High Availability: Sistem database transaksi dirancang toleran terhadap kegagalan (fault-tolerant) dengan uptime target mencapai 99.9%.

3. Analisis dampak jika salah satu kebutuhan tersebut tidak terpenuhi

  • Pencurian Data (Security): Kebocoran token transaksi ke pihak tak bertanggung jawab (Cyberattack).
  • Kerugian Finansial: Transaksi palsu/manipulasi saldo karena otorisasi tak terenkripsi.
  • Migrasi Massal: Kehilangan trust pengguna, eksodus pelanggan ke kompetitor.
  • Tuntutan Hukum: Pelanggaran regulasi perlindungan data pribadi (UU PDP).
  • Kelumpuhan Server (Availability): Sistem timeout dan gagal memproses transaksi saat jam sibuk.

EVALUASI ARSITEKTUR

Perlukah Semua Diagram?

Skenario Kasus (Soal 3.1): Tidak semua proyek perangkat lunak membutuhkan seluruh diagram UML. Menurut Anda, dalam proyek marketplace ini, apakah perlu menggunakan semua diagram yang telah Anda buat? Jelaskan alasannya!

Sangat Perlu (Wajib). Dalam konteks arsitektur marketplace yang melibatkan ekosistem eksternal multi-aktor (Payment gateway dan bank), penyusunan ketiga jenis diagram tersebut merupakan standar krusial untuk meminimalisir kesalahan interpretasi (miskomunikasi) antar tim pengembang:

  • Use Case Diagram: Kompas ruang lingkup bisnis. Biar tim paham siapa saja yang pakai aplikasi (User, Admin, Bank) dan apa batasannya.
  • Activity Diagram: SOP operasional. Biar tim Frontend & QA tahu rute sukses/gagal saat uji coba, agar pelanggan tidak "nyangkut" saat bayar.
  • Sequence Diagram: Penuntun teknis Backend. Memastikan urutan panggil API ("kurir pesan") ke Bank tepat, agar uang pelanggan tidak hangus karena error logika.
